# 확정 설계 결정사항 규칙

## 정책

코드 컨벤션·docs 로 일괄 판단이 어려운 항목은 Claude 가 임의 결정하지 말고 request.md "확정 설계 결정사항" 섹션에 **질문으로 나열**한다. 사용자가 답을 채운 뒤에 work 작성으로 진행. 답변 없는 항목이 남은 상태에서 Claude 는 work 파일을 작성하지 않는다.

## 질문 항목 식별 방법

하네스(이 플러그인)는 **어떤 카테고리가 "애매한지" 를 하드코딩하지 않는다.** 대신 Claude 가 초안을 쓸 때 아래 소스를 훑어보고, 해당 기능에 관련되면서 선택지가 필요한 항목을 직접 식별한다.

읽을 소스 (우선순위 순):
1. **프로젝트 루트 `CLAUDE.md`** — 프로젝트 규칙의 정답지
2. **`CLAUDE.md` 에서 링크된 `docs/*.md`** — 각 규칙의 상세 (`harness-config.json.harness_rules.enforced_docs` 로도 명시됨)
3. **기존 코드의 유사 기능** — 같은 도메인·유사 API 의 기존 구현

선택지가 필요한 항목이란:
- Y/N 결정 (예: "이 기능에 큐를 적용할지", "인증을 걸지")
- 값 선택 (예: "권한 레벨", "응답 형식", "페이지네이션 여부")
- 적용 범위 결정 (예: "새 util 을 전역에 둘지 도메인에 둘지")

**코드나 규칙 문서가 명확히 강제하는 항목은 질문하지 않는다.** 예: CLAUDE.md 가 "모든 repository 메서드는 try/catch 필수" 라고 단정하면 이는 결정사항이 아님 — 작성 시 그대로 따르면 됨.

## 질문 형식

request.md 본문 "확정 설계 결정사항" 섹션에 `[ ]` 체크리스트로 추가. 각 항목은 "어떤 결정인지 + 근거 섹션" 을 한 줄로:

```markdown
## 확정 설계 결정사항
- [ ] 인증 필요 여부 (Y/N) — 근거: CLAUDE.md 의 인증 관련 규칙 섹션:
- [ ] 큐 적용 여부 (Y/N) — 근거: CLAUDE.md 에 큐/메시지큐 관련 규칙 섹션이 있을 경우만:
- [ ] 성공 응답 형식 — 근거: CLAUDE.md 의 응답 형식 규칙:
```

> 특정 기능명(BullMQ, Redis, Kafka 등)을 이 문서에 직접 쓰지 말 것. "큐/메시지큐 관련 섹션이 있으면 그 규칙대로" 처럼 조건부로 참조. 프로젝트가 해당 기능을 안 쓰면 질문 자체가 생략됨.

사용자가 답(Y/N 또는 값)을 채운 뒤 Claude에게 work 작성을 요청 (예: `work 작성해줘` 같은 자연어) 하면 ④단계 (work.md 작성) 진입. ※ `작업 시작` / `/feature-implement` 는 work.md 가 이미 작성된 후의 ⑤단계 (구현 시작) 키워드라 여기선 사용하지 않음.

## 중단 조건

답변 없는 항목이 남아 있으면:
- Claude 는 work 파일을 작성하지 않는다.
- 사용자에게 "다음 항목에 답이 필요합니다: …" 로 재질의.
